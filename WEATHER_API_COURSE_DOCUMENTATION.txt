SECTION 1: COURSE ARCHITECTURE & OVERVIEW
================================================================================

1.1 COURSE PURPOSE
The "Weather API Course" is a hands-on, two-level curriculum designed to teach
software engineering students and professionals the core fundamentals of HTTP,
REST APIs, Python automation, asynchronous browser interactions (fetch), API
security, packaging, scheduled workflows (cron), webhooks, and Continuous
Integration (CI). 

Instead of teaching these dry subjects in isolation, the course tasks learners 
with building an increasingly sophisticated weather aggregation dashboard, moving 
from simple curl commands to a fully secure, scheduled, type-checked, and 
unit-tested enterprise automation system.

1.2 MULTI-SOURCE INTEGRATION & NORMALIZATION
One of the core architectural lessons of this course is "Response Normalization". 
Third-party APIs vary widely in response layouts, query parameter names, units, 
and error structures. Learners fetch weather data from three distinct providers:
- Open-Meteo: Requires geocoding lookups, returns data without keys, and uses
  WMO weather codes.
- OpenWeatherMap: Requires an API key (passed via query params) and uses standard
  JSON nested objects.
- WeatherAPI: Requires an API key and returns highly nested values including AQI.

The course forces learners to normalize these various responses into a single,
clean, unified data format:
{
    "source": str,        # "Open-Meteo" | "OpenWeatherMap" | "WeatherAPI"
    "temp_c": float,       # Temperature in Celsius
    "wind_kph": float,     # Wind speed in km/h
    "humidity": int,       # Relative humidity (%)
    "desc": str            # Cleaned up textual weather description
}

1.3 TESTING & TOOLING
The course starts with basic shell scripting, moves to standard Python requests, 
then scales to Postman/Newman for functional regression testing, Flask for webhooks, 
pytest with offline mocking for isolation, and GitHub Actions for automated 
CI pipelines.

SECTION 2: KEY ARCHITECTURAL & PROFESSIONAL DESIGN PATTERNS
================================================================================

The repository incorporates five core design patterns that mirror real-world,
enterprise production environments:

--------------------------------------------------------------------------------
PATTERN A: CLIENT-SIDE KEY MANAGEMENT (localStorage Settings UI in Level 1 / Phase 04)
--------------------------------------------------------------------------------
- PROBLEM: Frontend browser applications speaking directly to third-party APIs
  cannot securely read backend environment variables without a server. Hardcoding
  keys exposes secrets to the public.
- SOLUTION: A local "API Key Settings Panel" that captures keys securely via UI
  inputs and persists them directly to the browser's native `localStorage` vault.
  This allows developers to build, test, and run frontend weather dashboards
  locally or on static hosts (like Cloudflare Pages) completely client-side without
  needing databases or exposing credentials to Git.

--------------------------------------------------------------------------------
PATTERN B: STATIC CODE ANALYSIS & MODERN TOOLING (Level 2 / Phase 01)
--------------------------------------------------------------------------------
- PROBLEM: As a project scales from single-file scripts to multi-module packages,
  runtime type errors and stylistic inconsistencies grow.
- SOLUTION: Pyproject-based packaging with configuration files ('pyproject.toml')
  enforcing:
  * `ruff`: A lightning-fast linter and formatter that checks style violations,
    unreferenced variables, and formats imports automatically.
  * `mypy`: A strict static type-checker that checks python type hints (`strict = true`)
    to catch algorithmic bugs before execution.

--------------------------------------------------------------------------------
PATTERN C: OFFLINE API MOCKING (Level 2 / Phase 01)
--------------------------------------------------------------------------------
- PROBLEM: Automated test suites should never make live HTTP requests. If tests
  rely on real API endpoints, they are flaky (fail on network drops), slow, deplete
  API rate limits, and require secrets to be injected into CI runners.
- SOLUTION: Intercepting network traffic with Python's standard `unittest.mock.patch`
  library. Outbound calls to `requests.get` are hijacked, and pre-defined mock JSON 
  response payloads (with appropriate `.json()` return values and `raise_for_status`
  mocks) are returned. Tests execute 100% offline, reliably, and in milliseconds.

--------------------------------------------------------------------------------
PATTERN D: CRYPTOGRAPHIC WEBHOOK SIGNATURES (Level 2 / Phase 04)
--------------------------------------------------------------------------------
- PROBLEM: Webhook receiver endpoints are open to the public internet. Anyone who
  finds the URL can post fake or malicious weather payloads.
- SOLUTION: Cryptographic symmetric signature verification. The webhook sender
  signs the raw request body with a shared secret key using SHA256 HMAC. The Flask
  receiver reads the raw payload, calculates the HMAC signature locally, and compares
  it with the incoming `X-Hub-Signature-256` header. It uses `hmac.compare_digest` to
  prevent timing-attack vulnerabilities.

--------------------------------------------------------------------------------
PATTERN E: PRODUCTION EDGE PROXIES (Cloudflare Pages Functions in Level 1 / Phase 05)
--------------------------------------------------------------------------------
- PROBLEM: Real-world client-side production apps cannot rely on localStorage
  as user-facing keys. Secrets must stay on the server.
- SOLUTION: Serverless API Proxies (Backend-for-Frontend). Instead of calling
  external APIs directly from the browser, the frontend queries a local path:
  `fetch('/api/weather?city=Roseau')`.
  The platform (Cloudflare Pages) intercepts this path inside a `/functions` folder
  and executes a lightweight secure Edge function that injects the secret API key 
  from the platform's dashboard, performs the real fetch, and proxies the result 
  back to the user without ever exposing the key.

SECTION 4: LEVEL 1 (PHASES 00 TO 07) DETAILS & KEY CONCEPTS
================================================================================

--------------------------------------------------------------------------------
PHASE 00: curl BASICS
--------------------------------------------------------------------------------
- Focus: Proving APIs are plain HTTP over the wire.
- Key Tasks: Use `curl` to fetch raw text/JSON from endpoints like `wttr.in`,
  debug conversations via verbose flag `-v`, perform HTTP methods (GET, POST,
  PUT, DELETE) using `-X` against mock endpoints on `httpbin.org`, and inspect
  authorization codes (401 Unauthorized, 403 Forbidden).
- Core Learnings:
  * Headers supply metadata (`Accept: application/json`, `Content-Type: application/json`).
  * Body payloads are supplied with `-d '{"key": "val"}'`.
  * Status codes dictate outcome categories (2xx success, 4xx client errors, 5xx server errors).

--------------------------------------------------------------------------------
PHASE 01: PYTHON REQUESTS
--------------------------------------------------------------------------------
- Focus: Translating curl into programmable Python automation scripts.
- Key Tasks: Write a Python CLI tool using the `requests` library to fetch
  weather from Open-Meteo's coordinates-based forecasting API. Parse the nested
  JSON return object, manage connection/timeout errors, and accept command-line
  arguments with `sys.argv`.
- Core Learnings:
  * The `params` parameter in `requests.get(url, params=...)` handles safe URL parameter escaping.
  * `response.raise_for_status()` raises `requests.exceptions.HTTPError` on non-200 responses.
  * Adding explicit timeouts (`timeout=10`) is mandatory to prevent hanging processes.
  * Handling exceptions in blocks: `try ... except requests.exceptions.Timeout` is critical for robustness.

--------------------------------------------------------------------------------
PHASE 02: AUTHENTICATED APIs
--------------------------------------------------------------------------------
- Focus: Safely injecting secret keys and normalizing disparate response bodies.
- Key Tasks: Copy `.env.example` to `.env` (which is git-ignored). Retrieve API keys
  for OpenWeatherMap and WeatherAPI. Use `python-dotenv` to inject values into
  the process environment. Write separate fetchers and translate their varied responses
  into a unified standardized dictionary layout. Write logic to inspect rate-limiting
  response headers (e.g. `X-RateLimit-Limit`).
- Core Learnings:
  * Secrets must never be hardcoded or checked into source control.
  * Response normalization hides the complexity of diverse third-party APIs from
    downstream client layers.
  * API endpoints enforce limits; checking headers helps scripts adjust requests and avoid 429 errors.

--------------------------------------------------------------------------------
PHASE 03: MULTI-SOURCE AGGREGATION
--------------------------------------------------------------------------------
- Focus: Accelerating data retrieval using concurrency and preventing cascading failures.
- Key Tasks: Read input city, run Open-Meteo, OpenWeatherMap, and WeatherAPI fetches
  in parallel. Aggregate findings in a neat command-line table, outputting total latency.
- Core Learnings:
  * Parallel IO calls are initiated via `concurrent.futures.ThreadPoolExecutor`.
  * Using `as_completed(futures)` allows processing results as soon as they return, instead of waiting for the slowest.
  * A single source failing should be wrapped in `try-except` within the future loop, allowing the application to degrade gracefully and return partial results instead of crashing the process.

--------------------------------------------------------------------------------
PHASE 04: BROWSER FETCH
--------------------------------------------------------------------------------
- Focus: Transferring parallel execution onto a client-side browser dashboard.
- Key Tasks: Write an interactive webpage (`index.html`) using raw vanilla JS that
  asynchronously queries Open-Meteo, OpenWeatherMap, and WeatherAPI using native `fetch()`.
  Create a UI credentials settings panel persisting credentials in browser `localStorage`.
- Core Learnings:
  * Asynchronous JavaScript is managed using `async`/`await` primitives.
  * Parallel resolution is completed safely via `Promise.allSettled()`, which ensures
    failing requests do not abort successful siblings (unlike `Promise.all()`).
  * Cross-Origin Resource Sharing (CORS) security restricts how browsers request external domains.
  * Input parameters are escaped dynamically using the browser's `URLSearchParams` object.

--------------------------------------------------------------------------------
PHASE 05: DEPLOY YOUR VERSION
--------------------------------------------------------------------------------
- Focus: Setting up real CI/CD pipelines to a public hosting platform.
- Key Tasks: Extract the static frontend files and copy them to a separate personal
  git repository. Configure Cloudflare Pages (or similar) to deploy the repository
  on every main-branch push, utilizing basic security headers in a `_headers` config.
- Core Learnings:
  * Understanding static build environments (build commands, output directories, triggers).
  * Separation of concerns: Curriculum repository stays clean, and the actual application is isolated in its own deployment repo.

--------------------------------------------------------------------------------
PHASE 06: API SECURITY
--------------------------------------------------------------------------------
- Focus: Eliminating API secret leakage, secure credentials transportation, and rate-limits.
- Key Tasks: Build a custom static analyzer using Python's regular expressions (`re`)
  to scan source files for hardcoded secrets, pattern-matching patterns like `sk_live_...`
  or assignment keys (`api_key="..."`). Write an HTTP client checking for status 429 and
  reading `Retry-After` headers. Contrast URL query parameters authorization (unsafe due to log leaks)
  with Authorization header transport (safe).
- Core Learnings:
  * Web proxies and server logs capture complete URLs, making query-string API keys insecure. Header-based auth (`Authorization: Bearer <key>`) keeps credentials hidden inside encrypted HTTP packages.
  * Static analysis gates help developers catch hardcoded keys locally before git staging.

--------------------------------------------------------------------------------
PHASE 07: JSON DEEP DIVE
--------------------------------------------------------------------------------
- Focus: Data integrity, local caching, and safety validation of third-party JSON bodies.
- Key Tasks: Implement a local file-based cache in Python using `json.dump` and `json.load`.
  Build a robust deep-getter utility `safe_get` traversing dictionaries with unknown structures
  safely without raising `KeyError` failures. Implement lightweight schema type-validation.
- Core Learnings:
  * API caching limits roundtrips, saves bandwidth, and respects provider limits.
  * Navigating highly nested objects must be guarded; `data.get("a", {}).get("b", {}).get("c")`
    or recursive loops must be utilized.
  * Python's `json.dump` writes dictionary structures directly to disk, and `json.load` reads them back.

```python
"""
Level 1 — Phase 01: Python + requests (SOLUTION)
"""

import sys
import requests

WEATHER_CODES = {
    0: "Clear sky",
    1: "Mainly clear",
    2: "Partly cloudy",
    3: "Overcast",
    45: "Foggy",
    48: "Icy fog",
    51: "Light drizzle",
    61: "Light rain",
    71: "Light snow",
    80: "Rain showers",
    95: "Thunderstorm",
}


def get_weather(latitude: float, longitude: float) -> dict:
    """Fetch current weather from Open-Meteo. No API key needed."""
    url = "https://api.open-meteo.com/v1/forecast"
    params = {
        "latitude": latitude,
        "longitude": longitude,
        "current": "temperature_2m,wind_speed_10m,relative_humidity_2m,weather_code",
        "temperature_unit": "celsius",
        "wind_speed_unit": "kmh",
    }
    response = requests.get(url, params=params, timeout=10)
    response.raise_for_status()
    return response.json()["current"]


def get_weather_safe(latitude: float, longitude: float) -> dict | None:
    """Same as get_weather() but returns None on failure."""
    try:
        return get_weather(latitude, longitude)
    except requests.exceptions.Timeout:
        print("❌ Request timed out")
    except requests.exceptions.HTTPError as e:
        print(f"❌ HTTP error: {e.response.status_code}")
    except requests.exceptions.ConnectionError:
        print("❌ Connection error — check your network")
    except Exception as e:
        print(f"❌ Unexpected error: {e}")
    return None


def format_weather(data: dict, location: str = "Unknown") -> str:
    """Format weather data as a readable string."""
    code = data.get("weather_code", -1)
    condition = WEATHER_CODES.get(code, f"Code {code}")
    return (
        f"🌤  {location}\n"
        f"   Temperature : {data.get('temperature_2m', '?')}°C\n"
        f"   Wind        : {data.get('wind_speed_10m', '?')} km/h\n"
        f"   Humidity    : {data.get('relative_humidity_2m', '?')}%\n"
        f"   Condition   : {condition}\n"
    )


def post_weather_report(weather_data: dict, learner_name: str) -> None:
    """POST the weather data to httpbin.org to demonstrate sending data."""
    url = "https://httpbin.org/post"
    payload = {"learner": learner_name, "weather": weather_data}
    response = requests.post(url, json=payload, timeout=10)
    print(f"   POST status: {response.status_code}")
    print(f"   Echoed back: {response.json().get('json', {}).get('learner', '?')}")


if __name__ == "__main__":
    lat = float(sys.argv[1]) if len(sys.argv) > 1 else 15.30
    lon = float(sys.argv[2]) if len(sys.argv) > 2 else -61.39
    location = sys.argv[3] if len(sys.argv) > 3 else "Roseau, Dominica"

    data = get_weather_safe(lat, lon)
    if data is not None:
        print(format_weather(data, location))
        post_weather_report(data, learner_name="weather-course-learner")
    else:
        print("Failed to fetch weather")
        sys.exit(1)
```

"""
Level 1 — Phase 02: Authenticated APIs (SOLUTION)
"""

import requests
import os
from dotenv import load_dotenv

load_dotenv()


def get_weather_owm(city: str) -> dict | None:
    url = "https://api.openweathermap.org/data/2.5/weather"
    api_key = os.getenv("OPENWEATHER_API_KEY")

    if not api_key:
        print("❌ OPENWEATHER_API_KEY not found in .env")
        return None

    params = {"q": city, "appid": api_key, "units": "metric"}

    try:
        response = requests.get(url, params=params, timeout=10)

        if response.status_code == 401:
            print("❌ Invalid API key — check your .env")
            return None
        if response.status_code == 403:
            print("❌ Access forbidden — check your plan")
            return None
        if response.status_code == 429:
            print("❌ Rate limit exceeded — slow down")
            return None

        response.raise_for_status()
        check_rate_limits(response)

        d = response.json()
        return {
            "source": "OpenWeatherMap",

def scan_file_for_secrets(filepath: str) -> list[str]:
    p = Path(filepath)
    if not p.exists():
        return []
    text = p.read_text()
    findings = []
    for i, line in enumerate(text.splitlines(), start=1):
        stripped = line.strip()
        if not stripped or stripped.startswith("#"):
            continue
        if "os.getenv" in line or "os.environ" in line:
            continue
        for pat in SECRET_PATTERNS:
            if pat.search(line):
                findings.append(f"  L{i}: {stripped[:100]}")
                break
    return findings


def scan_directory(directory: str) -> None:
    print(f"Scanning {directory}...")
    root = Path(directory)
    total_findings = 0
    for py_file in root.rglob("*.py"):
        findings = scan_file_for_secrets(str(py_file))
        if findings:
            print(f"\n  ⚠️  {py_file}")
            for f in findings:
                print(f)
            total_findings += len(findings)
    if total_findings == 0:
        print("  ✅ Clean — no hardcoded secrets found")

    response = requests.get(
        "https://api.open-meteo.com/v1/forecast",
        params={
            "latitude": first["latitude"],
            "longitude": first["longitude"],
            "current": "temperature_2m,wind_speed_10m,relative_humidity_2m,weather_code",
            "wind_speed_unit": "kmh",
        },
        timeout=10,
    )
    response.raise_for_status()
    return normalize_current(response.json()["current"])

import os

load_dotenv(Path(__file__).parent / ".env")

from weather.sources import open_meteo  # noqa: E402


def get_today_forecast(city: str) -> dict:
    return open_meteo.fetch(city)


def build_message(weather: dict) -> dict:
    city = weather.get("city", "Roseau")
    humidity = weather.get("humidity", 0)
    desc = weather.get("desc", "").lower()
    temp = weather.get("temp_c", 0)

    if "rain" in desc or "drizzle" in desc or "shower" in desc or humidity > 85:
        text = f"Rain expected in {city}. Bring an umbrella. Humidity: {humidity}%."
    elif temp > 30:
        text = f"Hot day in {city}: {temp} deg C. Stay hydrated."
    else:
        text = f"Weather in {city}: {temp} deg C, {weather.get('desc', 'clear')}."

    return {"content": text}


def send_to_discord(payload: dict) -> int:
    url = os.getenv("DISCORD_WEBHOOK_URL")
    if not url:
        raise RuntimeError("DISCORD_WEBHOOK_URL is missing")

    response = requests.post(url, json=payload, timeout=10)
    return response.status_code


def main() -> int:
    city = sys.argv[1] if len(sys.argv) > 1 else "Roseau"
    weather = get_today_forecast(city)
    weather["city"] = city
    payload = build_message(weather)
    status = send_to_discord(payload)
    print(f"Sent to Discord: HTTP {status}")
    return 0 if 200 <= status < 300 else 1


if __name__ == "__main__":
    raise SystemExit(main())

"""
Level 2 / Phase 04 - Webhook receiver (solution).
"""

import logging
from datetime import datetime, timezone
import hmac
import hashlib

from flask import Flask, jsonify, request

app = Flask(__name__)
logging.basicConfig(level=logging.INFO, format="%(asctime)s [%(levelname)s] %(message)s")

# Shared secret key with the sender to cryptographically verify requests
WEBHOOK_SHARED_SECRET = b"super-secret-key-shared-with-sender"


def verify_signature(data: bytes, signature: str) -> bool:
    """Validate webhook authenticity using HMAC-SHA256."""
    expected_signature = hmac.new(
        WEBHOOK_SHARED_SECRET, 
        msg=data, 
        digestmod=hashlib.sha256
    ).hexdigest()
    # Use hmac.compare_digest to protect against timing attacks
    return hmac.compare_digest(expected_signature, signature)


@app.route("/webhook", methods=["POST"])
def webhook():
    payload = request.get_json(silent=True) or {}
    app.logger.info("Received webhook payload: %s", payload)
    app.logger.info("Headers: %s", dict(request.headers))
    return (
        jsonify(
            {
                "status": "received",
                "received_at": datetime.now(timezone.utc).isoformat(),
            }
        ),
        200,
    )


@app.route("/webhook-secure", methods=["POST"])
def webhook_secure():
    """
    Exposes a cryptographically secured webhook endpoint.
    Expects signature in 'X-Hub-Signature-256' header.
    """
    signature = request.headers.get("X-Hub-Signature-256")
    if not signature:
        app.logger.warning("Rejected webhook: Missing X-Hub-Signature-256 header")
        return jsonify({"error": "Missing signature"}), 401
        
    raw_body = request.get_data()
    if not verify_signature(raw_body, signature):
        app.logger.warning("Rejected webhook: Invalid cryptographic signature")
        return jsonify({"error": "Invalid signature"}), 403
        
    payload = request.get_json(silent=True) or {}
    app.logger.info("Secured Webhook Verified! payload: %s", payload)
    return (
        jsonify(
            {
                "status": "verified & processed",
                "processed_at": datetime.now(timezone.utc).isoformat(),
            }
        ),
        200,
    )


@app.route("/health", methods=["GET"])
def health():
    return jsonify({"status": "healthy"}), 200


if __name__ == "__main__":
    print("Starting webhook receiver on http://localhost:8080")
    print("In another terminal, run: ngrok http 8080")
    app.run(host="0.0.0.0", port=8080, debug=True)
